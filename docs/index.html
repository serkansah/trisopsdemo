PS Portal Nedir?What is PS Portal?

PS Portal, BT ekiplerinin müşteri veya kurum altyapılarını tek ekrandan izlemek, raporlamak ve yönetmek için kullandığı dahili altyapı yönetim portalıdır. Farklı sistemlerdeki bilgileri bir araya toplayarak mühendislerin bağlam değiştirmeden hızlı karar verebilmesini sağlar. PS Portal is an internal infrastructure management portal used by IT teams to monitor, report on, and manage customer or organizational infrastructure from a single screen. It brings together information from different systems so engineers can make fast decisions without switching context.

Neden Var?

Müşteri altyapısını takip etmek; vCenter, NetBackup, Zerto, OpManager, PAM gibi onlarca ayrı ürün konsolunu aynı anda açık tutmayı gerektiriyordu. Her platform kendi kullanıcı yönetimini, kendi URL'ini ve kendi arayüz diline sahipti. PS Portal bu dağınıklığı çözmek için ortaya çıktı:

  • Tüm sistemlerin durumu tek login ile görülür.
  • Farklı kaynaklardan gelen veriler normalize edilerek karşılaştırılabilir hale getirilir.
  • Müşteri toplantısına hazırlanmak için gereken raporlar dakikalar içinde üretilir.
  • VM operasyonları (başlat / durdur / yeniden başlat) onay akışıyla denetim altında tutulur.
  • Tüm aksiyonlar denetim günlüğüne yazılarak izlenebilirlik sağlanır.

Güvenlik Tasarım İlkeleri

  • En az yetki (RBAC): Her kullanıcıya yalnızca gerekli modüller ve sayfalar açılır; yetki dışındaki URL'ler 403 döner.
  • Çok faktörlü kimlik doğrulama (MFA): TOTP tabanlı doğrulayıcı uygulama desteği; admin tarafından zorunlu kılınabilir.
  • Denetim günlüğü: Tüm giriş, çıkış, ayar değişikliği ve kullanıcı aksiyonları silinemeyen kayıtlarla tutulur.
  • HTTPS zorunluluğu: SSL sertifikası yüklenmemişse topbar uyarı gösterir; üretimde HTTP devre dışıdır.
  • Şifreli depolama: Bağlantı şifreleri, SMTP şifresi ve API anahtarları veritabanında Fernet ile şifreli saklanır.
  • Tam felaket kurtarma yedeği: Otomatik yedekleme; veritabanı, SSL sertifikaları ve tüm uygulama kodunu tek arşivde saklar.
Bu dokümantasyon portalın tüm özelliklerini kapsar. Sol menüden istediğiniz konuya doğrudan atlayabilir ya da üstteki arama kutusunu kullanabilirsiniz.

Why Does It Exist?

Tracking customer infrastructure meant keeping dozens of separate product consoles open at once — vCenter, NetBackup, Zerto, OpManager, PAM. Each platform had its own user management, its own URL, and its own interface language. PS Portal was built to solve that sprawl:

  • The status of all systems is visible with a single login.
  • Data from different sources is normalized into a comparable form.
  • Reports needed to prepare for a customer meeting are generated in minutes.
  • VM operations (start / stop / restart) are kept under control through an approval workflow.
  • All actions are written to an audit log for traceability.

Technical Stack

The portal is designed to run on-premise with no cloud dependency. All data stays on the organization's own server.

Security Design Principles

  • Least privilege (RBAC): Each user only sees the modules and pages they need; unauthorized URLs return 403.
  • Multi-factor authentication (MFA): TOTP-based authenticator app support; can be enforced by admins.
  • Audit log: All logins, logouts, setting changes, and user actions are kept in tamper-proof records.
  • HTTPS enforcement: The topbar shows a warning if no SSL certificate is installed; HTTP is disabled in production.
  • Encrypted storage: Connection passwords, SMTP password, and API keys are stored Fernet-encrypted in the database.
  • Full disaster-recovery backup: Automatic backups store the database, SSL certificates, and all application code in a single archive.

Login Methods

The portal supports two authentication modes:

ModeHow it worksWhen it's used
Active DirectorySign in with your existing domain username and password. Group membership determines admin privileges.Production environment
Local (mock)Sign in with a fixed username/password set by the system administrator.Setup, testing, or when no AD connection is available
Recovery AccountEven if the AD connection drops, you can always sign in using the admin user and the recovery password set during installation.

First Login — Approval Process

Users who haven't yet been assigned a role see the "Your access request has been received" screen when they log in. This screen:

  • Confirms that the request has been recorded.
  • States that it needs approval from a system administrator.
  • Is visible to the system administrator under Admin Panel → Settings → Access Requests, where the request is assigned a role.
Browser SupportThe portal is tested on current versions of Chrome, Edge, and Firefox. Internet Explorer is not supported.

Password Reset

In Active Directory mode, password resets go through your own domain policy — the portal does not handle this. In local mode, contact your system administrator to change your password.

How It Works

  1. At least 2 characters are entered in the search box; automatic search starts after a 300 ms debounce.
  2. The vCenter snapshot database and OpManager device list are queried in parallel.
  3. Results are shown in two groups: vCenter VM and Physical Device (OpManager).

Performance

The OpManager device list is cached in memory for 5 minutes. This means a request isn't sent to the OpManager API on every keystroke — a fresh fetch only happens once the cache expires.

RVTools Archive History — Deleted Servers and Change Timeline

Search now also scans an index built from the daily RVTools Excel archive (roughly a year of history), in addition to live vCenter/OpManager data. This means servers that no longer exist in inventory can also be found — results appear in a separate "Archive Record / Deleted Servers" table, tagged "Deleted" or "Exists".

RVTools archive results — shown in a separate table below live vCenter results (server name/IP hidden)

Clicking the "History" button on either a live vCenter row or an archive row opens a chronological timeline of every change that server went through across the RVTools archive:

Server History — CPU/RAM changes, disk added/resized, power state, first/last seen dates
  • CPU / RAM change — logged with date and before/after value whenever vCPU or memory is increased or decreased.
  • Disk added / removed / resized — disks are tracked individually via RVTools' persistent "Disk Key" field.
  • Power state changed — poweredOn ↔ poweredOff transitions.
  • Disappeared from inventory / reappeared — a server is considered "deleted" once it no longer appears in the archive; if it later reappears, it's flagged "reappeared".
Out of scope: Zerto VRA system VMs (Z-VRA*) and temporary restore-test VMs with "_RESTORE" in their name are excluded from this history tracking, since they generate constant automatic disk/journal churn that would otherwise be noise.

The index is checked hourly by the app scheduler; a new RVTools file is processed automatically as soon as it lands (~1 second). The very first build scans the whole archive and can take a few minutes. Admin users can trigger a manual refresh from the index-status indicator in the top-right of the page.

Nasıl Hesaplanır

Her sunucunun RVTools arşivi boyunca kaydedilmiş RAM ve disk değişiklikleri (bkz. CMDB Arama → RVTools Arşiv Geçmişi) zaman serisine dönüştürülür ve üzerine doğrusal (lineer) regresyon uygulanır. Elde edilen aylık büyüme hızı, güncel değere eklenerek 1 ay / 3 ay / 6 ay / 1 yıl / 2 yıl / 3 yıl sonrası için projekte edilir.

"Sürekli Büyüyen" Sayılma Kriterleri

Yanıltıcı tek seferlik sıçramaları (ör. bir defalık toplu disk taşıma) trend olarak göstermemek için bir sunucunun bir metrikte (RAM veya disk) "büyüyor" sayılması için şu koşulların hepsi sağlanmalıdır:

  • En az 21 gündür arşivde izleniyor olmalı.
  • İlgili metrikte en az 2 ayrı değişiklik kaydedilmiş olmalı.
  • Yıllık büyüme en az 1 GB olmalı (gürültü/yuvarlama eşiği).
  • Doğrusal uyum kalitesi (R²) en az 0.25 olmalı — düzensiz/rastgele dalgalanan değerler hariç tutulur.

Zerto VRA sistem VM'leri ve adında "_RESTORE" geçen geçici restore-test VM'leri bu analize dahil edilmez (bkz. CMDB Arama → RVTools Arşiv Geçmişi'ndeki aynı hariç tutma kuralı).

Datacenter Bazlı Kapasite Tahmini

Üstteki tablo, her datacenter için RAM ve Disk sekmeleri arasında geçiş yaparak güncel toplam tahsis ve 6 ufuktaki (1 ay – 3 yıl) tahmini toplam ihtiyacı gösterir. Tahmin, o datacenter'daki tüm sunucuların güncel toplamına, yalnızca "büyüyor" sayılan sunucuların büyüme hızlarının toplamı eklenerek hesaplanır — sabit kalan sunucuların değişmeyeceği varsayılır.

Alt tablo, en az bir metrikte büyüme trendinde olan sunucuları büyüme hızına göre sıralı listeler. How It's Calculated

Each server's RAM and disk changes recorded across the RVTools archive (see CMDB Search → RVTools Archive History) are converted into a time series, and linear regression is applied to it. The resulting monthly growth rate is added to the current value to project 1 month / 3 months / 6 months / 1 year / 2 years / 3 years ahead.

Criteria for "Continuously Growing"

To avoid presenting a misleading one-off jump (e.g. a single bulk disk migration) as a trend, a server is only classified as "growing" for a given metric (RAM or disk) if all of the following hold:

  • Tracked in the archive for at least 21 days.
  • At least 2 separate changes recorded for that metric.
  • Annualized growth of at least 1 GB (noise/rounding threshold).
  • Linear fit quality (R²) of at least 0.25 — erratic/randomly fluctuating values are excluded.

Zerto VRA system VMs and temporary restore-test VMs with "_RESTORE" in their name are excluded from this analysis (same exclusion rule as CMDB Search → RVTools Archive History).

Datacenter-Level Capacity Forecast

The table at the top toggles between RAM and Disk tabs, showing each datacenter's current total allocation and projected total need at the 6 horizons (1 month – 3 years). The forecast is computed by adding the summed growth rate of only the "growing" servers to the datacenter's current total across all servers — non-growing servers are assumed to stay flat.

Continuously Growing Servers and Detail Chart

The lower table lists servers growing in at least one metric, sorted by growth rate. The badge in the Confidence column shows that metric's R² value (green = strong linear fit, yellow = moderate, red = weak). Clicking "Chart" on a row shows that server's historical actual values (solid line) together with its future projection (dashed orange line); the RAM/Disk tabs switch the metric shown.

The 2-3 year horizons in particular carry wide uncertainty since they're extrapolated from only ~1 year of archive data — verify with the relevant server owners before making firm budget/purchasing decisions.

Connection-Scoped Restriction

For modules that can have multiple connections — vCenter, NetBackup, OpManager, Zerto, Delinea — a role can be granted access to specific connections instead of the entire module. On the role edit screen, checking the relevant module box reveals a "Connection restriction (optional)" list beneath it.

Connection-Scoped Restriction — selected connections for the vCenter module (connection names hidden)
  • If none are selected — behavior is unchanged, the role gets access to every connection in the module.
  • If at least one connection is selected — the role is restricted to just that connection (or connections); unchecking the module box automatically clears the selections.
Scope: This restriction is enforced on the vCenter page, CMDB Search, and the AI Assistant — a scoped user sees only permitted connections' VMs in all three places. Other VM-related pages (Capacity Trend Analysis, FinOps, Operations, Dashboard totals) don't yet enforce this restriction.

VM Category-Scoped Restriction

In addition to connection scoping, VMs can be organized into categories — such as companies or departments — from Admin Panel → VM Categories, and a role can be granted access to specific categories only. This is used to split VMs by company and give each company's staff access only to their own VMs' information. When the vCenter module box is checked, a "VM category restriction (optional)" list appears just below the connection restriction list.

VM category restriction — directly below the connection restriction (connection and category names are sample data)
  • If none are selected — the role gets access to VMs in every category.
  • If at least one category is selected — the role is restricted to VMs in just that category (or categories).
  • When both connection and category restrictions are used together, a VM must pass both filters to be visible (intersection).
Scope: VM category restriction is enforced on the vCenter page and CMDB Search. CMDB's historical/deleted-server search (RVTools archive) is not yet covered.

AD Group Support

Instead of an individual username, you can enter an AD group's DN in the Members field (e.g. CN=InfraViewers,OU=Groups,DC=domain,DC=com). All members of that group are automatically included in the role. Group membership is cached at login time (1 hour TTL).

Microsoft only verifies identity; permissions in the portal are still decided by this app's own Role system.   • Diğer üç kural (art arda başarısız giriş, toplu VM işlemi, mesai dışı işlem) yalnızca giriş kilidiyle yetinir, aktif bir oturumu sonlandırmaz — düşük güvenilirlikli bir sinyalin (örn. mesai dışı meşru bir bakım işlemi) kullanıcıyı işinin ortasında atmaması için.
Yeni kurulan bir sistemde önce bir süre monitor modunda bırakıp olay geçmişini izlemek, hangi kuralların ortamınıza uygun olduğunu görmek için önerilir. Eşikler (deneme sayısı, pencere süresi, mesai saatleri) ayarlar sayfasından değiştirilebilir.

Sayfanın alt kısmındaki tablo, tetiklenen tüm olayları (kural, önem, kullanıcı, IP, detay, alınan eylem) listeler; kural veya kullanıcı adına göre filtrelenebilir. Otomatik eylem alınmamış bir olay satırında "Şimdi At" düğmesi bulunur — admin, monitor modunda bile, olayı gözden geçirdikten sonra elle o kullanıcının oturumunu anında sonlandırabilir.

Yurtdışı IP tabanlı engelleme bilinçli olarak eklenmedi — sunucular internete kısıtlı erişimli olabileceğinden, bir IP→ülke veritabanının yerel olarak gömülmesi gerekirdi. Bu, ileride ayrı bir değerlendirme konusu.

Monitored Rules

RuleWhen it firesSeverity
Repeated failed loginsSame username, a configured number (default 5) of failed login attempts within a configured window (default 5 min)Medium
Same user, logins from different IPsThe same username logs in successfully from more than one distinct IP within a short window (default 30 min)High
Mass VM operationsSame user performs a configured number (default 5) of VM operations (poweroff, delete, disk extend, etc.) within a short window (default 10 min)Medium
Off-hours VM operationA VM operation happens outside the configured business hours/days (default Mon–Fri 08:00–19:00, Turkey time)Medium
Permission-boundary probingSame user hits a page they don't have access to (403) a configured number of times (default 5) within a short window (default 5 min)High

If the same rule re-fires for the same user, a 30-minute cooldown throttles the mail/log entry so the inbox isn't spammed. This only throttles the notification — any automatic action below still re-applies on every trigger.

Monitor Mode and Automatic Actions

By default the feature runs in monitor mode: every rule is detected, logged, and emailed, but no user is automatically affected. If the "Automatic actions live" checkbox on the settings page is enabled:

  • Any triggered rule locks that user's login for a configured duration (default 5 min) — even with the correct password, they can't log in through any path, including Microsoft 365.
  • Only the two high-severity rules ("logins from different IPs", "permission-boundary probing") also immediately terminate the user's current session.
  • The other three rules (repeated failed logins, mass VM operations, off-hours operations) only apply the login lock, not an active-session kill — so a lower-confidence signal (e.g. legitimate off-hours maintenance) doesn't yank someone out mid-task.
On a newly set up system, leave it in monitor mode for a while first and watch the event history to see which rules fit your environment before going live. Thresholds (attempt counts, window lengths, business hours) can all be changed on the settings page.

Event History and Manual Action

The table at the bottom of the page lists every triggered event (rule, severity, user, IP, detail, action taken); filterable by rule or username. A row without an automatic action shows a "Kick Now" button — an admin can review the event and manually terminate that user's session immediately, even while in monitor mode.

Out of Scope

Foreign-IP-based blocking was deliberately not included — since servers may have restricted internet access, this would require embedding a local IP→country database. This remains a separate consideration for the future.

Before Expiry — Header Warning Banner

At 90, 60, 30, 7, 3, and 1 days before the end date, a warning banner appears on top of every page. It's amber at 90/60 days and turns red at 30 days and below. Each threshold is independently dismissible with ✕ — dismissing one doesn't suppress the next (more urgent) one.

Once Expired

For 7 days past the end date (grace period), the app keeps working normally — only the banner becomes impossible to dismiss. Once that grace period also ends (or the user-count limit is exceeded), access to every page except /license is blocked — including admin accounts, with no exception. This is intentional: otherwise a customer could defeat the entire license check by simply making everyone an admin. Turning the server clock back doesn't "un-expire" the license — the system stores the latest date it has ever seen for this install, and that date never moves backward; expiry checks always use whichever is later, "today" or that stored date.

Güvenlik MimarisiSecurity Architecture

PS Portal, kurumsal altyapı bilgilerine erişim sağladığından derinlemesine savunma (defence-in-depth) ilkesiyle tasarlanmıştır. Kimlik doğrulama, yetkilendirme, şifreleme ve denetim katmanları birbirini tamamlar. Because PS Portal grants access to sensitive corporate infrastructure information, it's designed on defence-in-depth principles. Authentication, authorization, encryption, and audit layers complement one another.

Token Blacklist (Session Invalidation)

When a user logs out, or an admin force-terminates a session, the token is invalidated even if it hasn't expired yet. A two-layer mechanism handles this:

LayerTriggerScope
Token hash blacklistUser logs outThat single token is invalidated; its SHA-256 hash is written to the token_blacklist table
Per-user revokeAdmin uses "Revoke Session"All of that user's active sessions are invalidated; a timestamp is written to the user_revocations table

Two checks run on every token verification: (1) is it in the hash blacklist? (2) was the token issued before the user's revoke timestamp? Both checks run from memory (an in-memory set/dict) — SQLite is written to only for persistence. Unexpired records are reloaded into memory when the server restarts.

Only the SHA-256 hash is kept in the blacklist; token contents are never stored. Expired entries are cleaned up periodically.

Authorization (RBAC)

Every user is assigned at least one role; roles determine which pages they can access. The system enforces a two-layer check:

  • Middleware layer — On every request, the session token is verified and the user's permission for the relevant module is checked.
  • Endpoint layer — Admin operations use the require_admin dependency; non-admin users get a 403.

Admin users can access all pages; no role check is performed for them. In partial-access scenarios (e.g. vCenter only), the user is shown only the relevant modules.

How It Works

The assistant uses a two-layer architecture:

  1. Rule-based dispatch — the question is first matched against known categories (VM count, licenses, quotes, etc.). If there's a match, the answer is generated directly from a database query; no LLM is used.
  2. LLM fallback — if there's no rule match, relevant infrastructure data is added to the question as context and sent to the configured LLM.
Rule-based answers use real-time database data, so they contain no hallucination. In the LLM fallback, context injection is performed, but the accuracy of the answer depends on the model.

  • En az bir kategori seçilirse — bot yalnızca o kategori(ler)deki VM'ler hakkında bilgi verir; diğer VM'lerin varlığından bahsetmez.

Güvenlik

Her Teams isteği, HMAC-SHA256 imzasıyla doğrulanır. İmza, Teams'in sağladığı Base64 kodlu Security Token kullanılarak Authorization: HMAC <base64sig> başlığında gönderilir. İmza doğrulanamazsa istek 401 ile reddedilir.

Ağ erişimi: Microsoft 365 tabanlı Teams kullanıyorsanız portalın internet üzerinden erişilebilir olması gerekir. Şirket içi ağda yayın yapıyorsanız Cloudflare Tunnel veya reverse proxy kullanabilirsiniz. Şirket içi Teams Server'da iç ağ adresi yeterlidir.

How It Works

  1. An Outgoing Webhook is created in Teams, with the portal address entered as the Callback URL.
  2. In the channel, the bot name is tagged with @ and a question is typed.
  3. Teams sends an HMAC-SHA256-signed POST request to the portal.
  4. The portal verifies the signature, forwards the question to the chatbot engine, and returns the answer to Teams.
The Teams Outgoing Webhook enforces a 5-second response time limit. Rule-based queries (VM count, license list, etc.) are answered within this time. Complex questions requiring the LLM may time out.

Setup Steps

  1. Open the Teams channel → click the ··· menu in the top right → select Workflows or Outgoing Webhooks.
  2. Click the "Create Outgoing Webhook" button.
  3. Enter a bot name (e.g. PS Portal Bot) and enter the portal address as the Callback URL:
    https://<server-address>/ai/teams
  4. Click Create — Teams will give you a Security Token. Copy this token.
  5. Open Portal Admin Panel → Integrations → Microsoft Teams.
  6. Paste the copied Security Token and click Save.
  7. Test it by typing @PS Portal Bot how many vms are there? in the Teams channel.

Scope Restriction

The Teams bot relies on a single security token shared per channel, not any individual user session — so its answers can't be restricted per person. Even so, the "VM Categories It Can See" box on the Admin Panel → Microsoft Teams page lets you limit the bot's VM inventory answers to specific VM categories only.

  • If no category is selected — anyone in the channel can query the full VM inventory (default behavior).
  • If at least one category is selected — the bot only answers about VMs in that category (or categories); it won't mention the existence of other VMs.

Security

Every Teams request is verified with an HMAC-SHA256 signature. The signature is sent in the Authorization: HMAC <base64sig> header, computed using the Base64-encoded Security Token provided by Teams. If the signature can't be verified, the request is rejected with a 401.

Network access: If you're using Microsoft 365-based Teams, the portal must be reachable over the internet. If you're serving on an internal network, you can use Cloudflare Tunnel or a reverse proxy. For on-premises Teams Server, an internal network address is sufficient.

The bot automatically strips the @ tag and HTML mention tags before generating a response; it works with the question text only.

Yedekleme & Felaket KurtarmaBackup & Disaster Recovery

PS Portal tüm verisini tek bir SQLite veritabanında tutar. Yedekleme modülü bu veritabanını, SSL sertifikalarını ve sistem dosyalarını tek bir arşivde toplar; Ubuntu dahil sunucu tamamen çökse bile birkaç komutla sistemi geri getirebilirsiniz. SQLite boş sayfa ve tekrar eden yapı içerdiğinden gzip ile %85–92 küçülür. 6 MB veritabanı tipik olarak ~600 KB arşiv üretir. Uygulama kodu ve config dosyaları da küçük olduğundan toplam arşiv genellikle 5–15 MB olur. Bu normaldir; veri eksiksizdir.

Portal'dan İndirme (Şifreli)

Bir yedek arşivi host/.env içindeki SECRET_KEY ve ENCRYPTION_KEY'i düz metin taşıdığı için, tek başına indirilebilir bir dosya tüm sistemin ele geçirilmesi için yeterli olabilir. Bu yüzden Yedekleme sayfasındaki "İndir" düğmesi, arşivi tarayıcıya göndermeden önce sizin belirlediğiniz bir parolayla şifreler.

Yedek indirme — şifreleme parolası isteniyor
  1. Yedek satırındaki "İndir" düğmesine tıklayın.
  2. En az 12 karakterlik bir şifreleme parolası girin — bu parola hiçbir yerde saklanmaz, yalnızca o an bellekte kullanılır.
  3. İndirilen dosya .tar.gz.enc uzantılıdır; ekranda bir kerelik bir Python çözme script'i (decrypt_backup.py) gösterilir — kopyalayıp saklayın.
  4. Çözmek için: pip install cryptography ardından python3 decrypt_backup.py backup_XXXXXXXX.tar.gz.enc — parolanızı soracaktır.
Şifreleme parolasını kaybederseniz arşivi kimse çözemez — parolayı bir şifre yöneticisinde saklayın. Bu koruma yalnızca "İndir" düğmesiyle indirme anını kapsar; sunucudaki Backup/ dizinindeki arşiv dosyaları diskte şifresiz durur (SFTP/SMB ile uzak aktarım da şifresiz gider) — bu, host'un kendisinin güvenliğine bağlıdır.

Her indirme, denetim kaydına düşer ve admin'e mail ile bildirilir (kim, hangi dosya, hangi IP'den). Beklenmedik bir bildirim alırsanız SECRET_KEY ve ENCRYPTION_KEY'i hemen döndürün.

Why Does the Backup Size Look Small?

Because SQLite contains empty pages and repetitive structure, gzip shrinks it by 85–92%. A 6 MB database typically produces a ~600 KB archive. Since the application code and config files are also small, the total archive is usually 5–15 MB. This is normal; the data is complete.

Downloading From the Portal (Encrypted)

Since a backup archive carries SECRET_KEY and ENCRYPTION_KEY in plaintext inside host/.env, a single downloadable file could be enough to compromise the whole system on its own. That's why the "Download" button on the Backup page encrypts the archive with a password you provide before it ever leaves the server.

Backup download — prompting for an encryption password
  1. Click "Download" next to a backup row.
  2. Enter an encryption password of at least 12 characters — this password is never stored anywhere, it's only used in memory for that one operation.
  3. The downloaded file has a .tar.gz.enc extension; a one-time Python decrypt script (decrypt_backup.py) is shown on screen — copy and keep it.
  4. To decrypt: pip install cryptography, then python3 decrypt_backup.py backup_XXXXXXXX.tar.gz.enc — it will prompt for your password.
If you lose the encryption password, no one can decrypt the archive — keep it in a password manager. This protection only covers the moment of download via the "Download" button; the archive files sitting in the server's Backup/ directory remain unencrypted on disk (and any SFTP/SMB remote transfer is unencrypted too) — that relies on the security of the host itself.

Every download is recorded in the audit log and emailed to admins (who, which file, from what IP). If you get an unexpected notification, rotate SECRET_KEY and ENCRYPTION_KEY immediately.

Even though backup files are created from inside the container (as root), the system automatically applies the same ownership as the Backup/ directory. This avoids permission issues when accessing them via SFTP or a file manager.

Backup Verification

To confirm a backup is actually usable for recovery, every backup row on the Backup Management screen has a "Verify" button. It opens the backup archive and tests the database inside it — the live system is never touched.

Backup verification results — all checks passing
  • Checks that the archive (.tar.gz) opens without corruption.
  • Confirms data/secrets.db and host/.env are present in the archive.
  • Extracts the database to a temporary directory and runs SQLite's PRAGMA integrity_check.
  • Checks that the expected tables (connections, app_settings, roles, etc.) exist.
  • Counts the rows in the connections table and shows the count in the result.
Verification is fully read-only — the database is only extracted to a temporary directory and checked there; the live secrets.db is never opened. Running it before you need to rely on a backup is recommended practice.

Full System Recovery (Ubuntu from Scratch)

The RESTORE.md file inside the archive contains an exact copy of these steps. You can open and read the archive even without internet access or access to this page.
  1. Install Docker on Ubuntu 
    curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER
newgrp docker
  2. Prepare the directory and extract the backup 
    sudo mkdir -p /opt/infra-portal
sudo chown $USER:$USER /opt/infra-portal
cd /opt

# copy the backup file here (scp / USB / SFTP)
tar xzf backup_XXXXXXXX_XXXXXX.tar.gz
  3. Place the host files 
    cp -r host/. /opt/infra-portal/

    .env, docker-compose.yml, nginx/, app/, and all other files are put in place in this step.

  4. Create and populate the Docker volume 
    docker volume create portal_data
docker run --rm \
  -v portal_data:/data \
  -v $(pwd)/data:/src \
  alpine sh -c "cp -rp /src/. /data/"

    Moves the database and SSL certificates into the Docker volume.

  5. Create the backup directory and start the system 
    mkdir -p /opt/infra-portal/Backup
cd /opt/infra-portal
docker compose up -d
  6. Check status 
    docker compose ps
docker logs infra-portal-app-1 --tail 30

    It should be reachable at https://SERVER_IP within a few seconds.

Restoring Only the Database (While the System Is Running)

If data loss occurred while the system was up, a full recovery isn't necessary. The Restore button on the Admin Panel → Backup page puts the database and SSL certificates back in place. Restart the container afterward for changes to fully take effect.

From the command line:

# 1. Extract the backup
tar xzf backup_XXXXXXXX_XXXXXX.tar.gz data/secrets.db

# 2. Copy into the running container
docker cp data/secrets.db infra-portal-app-1:/data/secrets.db

# 3. Restart the container
cd /opt/infra-portal && docker compose restart app
ENCRYPTION_KEY is critical — if the ENCRYPTION_KEY in the .env file is lost, all connection credentials in the database become permanently unrecoverable. This key is preserved in the backup archive (host/.env). For extra security, also keep .env stored separately in a secure location (a password manager, an encrypted USB drive).
